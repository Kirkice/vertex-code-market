# 着色管线

## 文档定位

本文档覆盖从场景数据到最终像素的渲染流程，包括 Forward/Deferred Shading、光源分配、抗锯齿和 Shader 管理。适用于需要设计渲染管线、选择着色方案或排查渲染问题的开发者。

## 一句话结论

光栅化擅长处理当前视图的屏幕覆盖区域，但不擅长直接回答任意两个空间点之间的可见性问题；因此全局光照通常需要额外的阴影、缓存、屏幕空间或体积数据通道。

## 核心概念

### 基本流程

```text
应用/场景数据
  → 顶点处理
  → 图元装配与裁剪
  → 光栅化
  → 深度/模板测试
  → 像素着色
  → 混合与后处理
```

光栅化擅长处理当前视图的屏幕覆盖区域，但不擅长直接回答任意两个空间点之间的可见性问题。因此全局光照通常需要额外的阴影、缓存、屏幕空间或体积数据通道。

### Forward、Deferred 与混合路径

#### Forward Shading

物体绘制时直接完成材质和光照计算。

- **优点**：透明物体和 MSAA 处理自然
- **缺点**：多个光源可能导致重复计算

#### Deferred Shading

先把材质、法线、深度等写入 G-buffer，再在后续 Pass 中统一计算光照。

- **优点**：光照与几何解耦，适合大量动态光源
- **代价**：额外 G-buffer 带宽和显存，并增加透明物体、MSAA 和移动端处理复杂度

#### 路径选择依据

需要综合考虑：

- 透明比例
- G-buffer 格式和带宽
- MSAA
- 材质复杂度
- 目标平台的 tile-based GPU 特性
- 后续是否需要 SSR、屏幕空间 GI 等后处理

### 光源分配

当光源很多时，不应让每个像素遍历全部光源。

- **Tiled Shading**：屏幕划分为二维 tile，每个 tile 保存影响它的光源列表。
- **Clustered Shading**：在 tile 上沿深度继续划分 cluster，更准确地处理空间范围。

典型流程：

1. 根据光源体积与摄像机视锥计算影响范围。
2. 为 tile 或 cluster 建立光源索引列表。
3. 像素着色时只遍历对应列表。

主要成本包括列表构建、索引缓冲区、深度划分和列表溢出处理。

### 抗锯齿与采样

锯齿来自连续图像被有限分辨率采样。抗锯齿的本质是改进空间、时间或几何采样。

- **MSAA**：增加几何覆盖采样，适合几何边缘。
- **FXAA/MLAA**：屏幕后处理，成本低但可能损失细节。
- **TAA**：利用历史帧和运动信息提高采样密度，但依赖稳定的运动矢量和历史有效性。
- **超采样或重建**：以更高采样率或时空重建换取质量。

实时 GI、反射和阴影也属于采样问题，因此抖动、重投影、历史重用和滤波思想会在后续章节反复出现。

### Shader 与渲染状态管理

Shader 管理包括变体数量、资源绑定、常量缓冲区更新频率、Render Target、深度/混合状态、Pass 顺序和资源生命周期。

常见风险：

- 变体爆炸导致编译和包体膨胀。
- 每个材质独立切换状态造成 CPU 提交成本。
- 不必要的中间 RT 导致带宽和显存增加。
- 资源生命周期不清导致同步、泄漏或未初始化数据。

## 工作原理

### G-buffer 设计

G-buffer 通常包含：

- 深度/法线
- 反照率/金属度/粗糙度
- 运动矢量（用于 TAA）
- 自定义数据（如 AO、自发光）

设计原则：

- 使用最小必要格式（如 RGB10A2 而非 RGBA32F）
- 合并相关数据减少 RT 数量
- 考虑 tile-based GPU 的带宽特性

### Tiled/Clustered Shading

#### Tiled Shading

- 屏幕划分为固定大小的 tile（如 16x16）
- 每个 tile 维护一个光源索引列表
- 像素着色时只遍历对应 tile 的光源

#### Clustered Shading

- 在 tile 基础上沿深度划分 cluster
- 更准确地处理光源的空间范围
- 适合大场景和远距离光源

### 抗锯齿技术对比

| 技术 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| MSAA | 几何覆盖采样 | 高质量，无模糊 | 成本高，不支持后处理 |
| FXAA | 边缘检测+模糊 | 成本低，后处理 | 可能损失细节 |
| TAA | 历史帧重用 | 高质量，支持后处理 | 需要运动矢量，可能鬼影 |
| SMAA | 边缘检测+模式匹配 | 平衡质量和成本 | 实现复杂 |

## 关键数据结构

### G-buffer 布局示例

| RT | 格式 | 内容 |
|---|---|---|
| RT0 | RGB10A2 | 反照率 + 金属度 |
| RT1 | RGBA16F | 法线 + 粗糙度 |
| RT2 | RG16F | 运动矢量 |
| Depth | D32F | 深度 |

### 光源列表结构

```text
LightList {
  uint count;
  uint indices[MAX_LIGHTS_PER_TILE];
}
```

### Cluster 索引结构

```text
ClusterIndex {
  uint lightOffset;
  uint lightCount;
}
```

## 开发者如何判断

### 选择 Forward 还是 Deferred

| 条件 | 推荐 |
|---|---|
| 大量透明物体 | Forward |
| 需要 MSAA | Forward |
| 大量动态光源 | Deferred |
| 复杂材质 | Deferred |
| 移动端 tile-based GPU | Forward+ 或轻量 Deferred |
| 需要屏幕空间后处理 | Deferred |

### 选择 Tiled 还是 Clustered

| 条件 | 推荐 |
|---|---|
| 光源分布均匀 | Tiled |
| 光源距离差异大 | Clustered |
| 大场景 | Clustered |
| 简单场景 | Tiled |

### 选择抗锯齿技术

| 条件 | 推荐 |
|---|---|
| 性能受限 | FXAA |
| 需要高质量 | TAA |
| 不支持后处理 | MSAA |
| 平衡质量和成本 | SMAA |

## 实现方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| Forward | 透明自然，MSAA 支持 | 多光源成本高 | 简单场景，透明物体多 |
| Deferred | 光照与几何解耦 | G-buffer 带宽，透明复杂 | 复杂场景，大量光源 |
| Forward+ | 结合两者优点 | 实现复杂 | 现代引擎，灵活需求 |
| Tiled Shading | 简单，适合均匀光源 | 深度处理差 | 简单场景 |
| Clustered Shading | 深度处理准确 | 内存和构建成本 | 复杂场景 |
| MSAA | 高质量几何边缘 | 成本高，不支持后处理 | 高质量需求 |
| TAA | 高质量，支持后处理 | 需要运动矢量，可能鬼影 | 现代渲染 |

## 常见问题与排查

| 现象 | 可能原因 | 验证方式 | 处理方向 |
|---|---|---|---|
| 透明物体排序错误 | 未正确排序透明物体 | 检查渲染顺序 | 实现透明排序 |
| G-buffer 带宽高 | RT 格式过大 | 查看 RT 格式 | 使用压缩格式 |
| 光源列表溢出 | 单 tile 光源过多 | 查看光源分布 | 增加列表大小或优化光源 |
| TAA 鬼影 | 运动矢量不准确 | 检查运动矢量 | 修正运动矢量计算 |
| Shader 变体过多 | 关键字组合爆炸 | 查看变体数量 | 减少关键字，使用动态分支 |
| 状态切换频繁 | 材质未批处理 | 查看 Draw Call | 实现材质批处理 |
