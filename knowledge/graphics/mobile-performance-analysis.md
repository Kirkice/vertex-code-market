# 移动端图形性能分析方法论

## 1. 适用范围

本文档用于沉淀移动端游戏与渲染 Demo 的图形性能分析方法，重点关注以下问题：

- 帧率不稳定或平均帧率偏低
- 运行一段时间后帧率下降
- 发热明显、耗电过快
- GPU 带宽压力过高
- 某些场景、物体、后处理或 Shader 导致性能突降

本文聚焦性能分析思路、指标体系、工具链与常见优化方向，适合作为 [`graphics-optimization`](src/core/prompts/sections/graphics-knowledge/mobile-performance-analysis.md) 相关知识输入的一部分。

---

## 2. 首先看哪些指标

拿到一个游戏或渲染 Demo 后，建议先看三类核心指标：

1. 帧率
2. 带宽
3. 发热与耗电

### 2.1 帧率

重点不只看平均 FPS，还要看：

- 平均帧率
- low frame 表现
- 帧率波动幅度
- 特定场景切换时的抖动

如果平均帧率尚可，但 low frame 很差，通常说明存在阶段性热点、同步抖动或个别 Pass / Draw 异常重。

### 2.2 带宽

带宽过高不仅会影响当前帧耗时，还会带来两个连锁问题：

- 设备发热增加
- 长时间运行后触发降频

因此带宽分析不能只看单帧，还要看持续运行后的趋势。

### 2.3 发热与耗电

移动端性能分析必须观察长稳表现，而不是只看冷机短时间测试。

建议进行自动运行或固定路径回放，持续 10 分钟到 1 小时，重点观察：

- 机身温度变化
- 帧率是否持续下滑
- 功耗是否异常
- 是否出现热降频后的性能回落

---

## 3. 推荐分析流程

建议按以下顺序推进：

```text
整体观察 → 判断 CPU / GPU 瓶颈 → 定位热点来源 → 选择专项优化方向 → 回到真机验证长稳表现
```

拆开来看：

1. 先跑起来，看 FPS、带宽、发热、耗电
2. 判断是 CPU 瓶颈还是 GPU 瓶颈
3. 如果偏 GPU，再继续拆：
   - draw call
   - mesh 顶点 / 面数
   - shader 复杂度
   - overdraw
   - RT 切换
   - 贴图与 RT 带宽
4. 做针对性优化后，重新回到真机测试

---

## 4. 如何区分 CPU 瓶颈与 GPU 瓶颈

### 4.1 CPU 瓶颈

可通过 Unity Profiler 一类工具查看耗时堆栈，判断主线程、渲染线程或脚本逻辑是否存在明显热点。

常见特征：

- C# 方法栈耗时明显
- 主线程或渲染线程时间高
- 降低渲染复杂度后收益不明显

典型处理方式：

- 定位具体耗时函数
- 优化脚本逻辑、资源加载、组件更新、批量操作方式
- 检查不必要的每帧计算与 GC

> 注：这类分析通常要求使用开发包或带 profiling 能力的构建。

### 4.2 GPU 瓶颈

如果降低画面复杂度、关闭部分渲染内容后帧率明显回升，则通常偏向 GPU bound。

GPU 侧常见热点来源包括：

- draw call 过多
- mesh 顶点 / 面数过高
- shadow 或后处理过重
- shader 指令过多
- overdraw 高
- RT 切换频繁
- 贴图过多、过大或 RT 太多导致带宽过高

## 5. GPU 侧专项分析方法

### 5.1 draw call 过多

可通过 FrameDebugger 或任意截帧工具观察 Draw Call 数量。

经验判断：

- 新一代移动设备对 draw call 相比旧设备不那么敏感
- 但数量仍不能过高，尤其在低端或老机型上仍需控制

常见优化方向：

- 合并 Mesh
- 合并贴图
- 使用 SRP Batching
- 使用 Draw Mesh Instancing
- 减少材质变体与状态切换

### 5.2 模型面数、顶点数或 Shadow 复杂度过高

如果某个物体被显示后帧数明显变化，可优先怀疑：

- 网格顶点数过高
- 面数过高
- UV 或额外顶点属性负担大
- 阴影相关绘制过重

排查方法：

- 通过显示 / 隐藏物体做对照测试
- 在真机截帧工具中观察对应 Draw 的耗时或时钟趋势
- 回到资源侧检查 Mesh 复杂度与数据布局

如果确认不是 Mesh 复杂度，通常就要继续怀疑 Shader 成本。

### 5.3 Shader 复杂度过高

对 Shader 的分析建议结合离线编译工具，例如 Mali Offline Compiler 一类工具，对目标机型做离线编译，观察：

- 指令数量
- 寄存器压力
- 纹理采样数量
- 不同精度对性能的影响

然后做针对性优化，例如：

- 降低指令复杂度
- 减少纹理采样
- 使用更合适的精度
- 简化分支与循环

### 5.4 overdraw 过高

高 overdraw 在移动端通常来自：

- 半透明物体太多
- 粒子过多
- 大面积叠层 UI
- 不必要的屏幕空间特效

常见优化方向：

- 减少半透明使用
- 控制粒子覆盖面积与数量
- 用 Alpha Clip 替代部分透明混合
- 尽量减少无效像素着色

需要注意：Alpha Clip 虽能降低部分像素开销，但也要结合具体 GPU 架构看 Early-Z 与裁剪副作用，不能机械套用。

### 5.5 RT 切换过多

RT 切换问题通常出现在：

- 管线设计不规范
- 后处理 Pass 组织不合理
- 临时 RT 使用过多

排查方法：

- 通过截帧工具观察 Render Target 切换频率
- 回到代码实现检查是否能合并 Pass
- 检查是否存在无意义的中间 RT

常见优化方向：

- 减少不必要的 RT 切换
- 合并后处理阶段
- 精简中间缓冲
- 重新梳理 Render Pass 依赖

### 5.6 贴图过多、过大或 RT 太多导致带宽高

这类问题通常可在截帧工具中观察到：

- 贴图格式
- 尺寸
- 使用张数
- Render Target 数量与格式

常见优化方向：

- 贴图压缩
- 选择合适的压缩格式
- 控制纹理尺寸
- 减少同时使用的纹理数量
- 控制 RT 数量和位宽

对于 ASTC，一般还要按内容类型选择合适 block size：

- 高频细节内容不要压得过猛
- RGB 与 Alpha 如果允许拆分，可对 Alpha 采用更激进压缩策略

## 6. 工具链建议

### 6.1 Unity Profiler

适合做：

- CPU / GPU 大方向归因
- 脚本热点分析
- 主线程与渲染线程耗时查看

### 6.2 FrameDebugger

适合做：

- Draw Call 数量观察
- Pass 组织检查
- 材质与渲染顺序排查

### 6.3 截帧工具

适合做：

- 观察单个 Draw / Pass 的成本趋势
- 检查 RT 切换
- 检查纹理、RT、Shader、渲染状态

### 6.4 离线编译工具

例如 Mali Offline Compiler 一类工具，适合做：

- Shader 离线分析
- 针对目标机型查看理论热点
- 为精度、采样与指令优化提供依据

---

## 7. 真机与回放的判断原则

在移动端图形分析中，需要特别强调：

- 真机结果是主要依据
- PC 回放或离线回放更适合看趋势
- 回放数值不能直接作为最终定量标准

这意味着：

- 可以用回放快速筛问题
- 但最终结论仍要回到目标设备验证
- 特别是热降频、带宽瓶颈、长时间帧率回落，必须在真机观察

---

## 8. 一个实用的排查清单

当帧率偏低时，可以依次问自己：

1. 是 CPU 瓶颈还是 GPU 瓶颈
2. 是否 draw call 过多
3. 是否某些模型顶点或面数过高
4. 是否某些 shadow / 后处理 Pass 太重
5. 是否 Shader 过于复杂
6. 是否 overdraw 高
7. 是否 RT 切换过多
8. 是否贴图过多、过大或 RT 太多导致带宽高
9. 是否长时间运行后因发热触发降频

这个清单适合作为性能初筛入口，而不是一开始就陷入单个 Shader 或单个 Pass 的微观调优。

---

## 9. 可沉淀为 Graphics 知识库的经验结论

1. 移动端性能分析不能只看冷机短测，必须看长稳表现
2. 带宽问题不仅影响当前帧，还会通过发热与降频影响后续性能
3. GPU 瓶颈需要继续拆成 draw call、mesh、shader、overdraw、RT、贴图带宽等具体问题
4. 某些问题更适合通过显示 / 隐藏物体的 A/B 测试快速缩小范围
5. 真机截帧看趋势、Profiler 看归因、离线编译看 Shader 成本，三者应配合使用
6. PC 回放更适合辅助判断趋势，不适合作为移动端最终性能定量标准
