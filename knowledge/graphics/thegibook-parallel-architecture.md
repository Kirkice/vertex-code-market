# 并行处理器架构

## 文档定位

本文档覆盖 CPU 与 GPU 的架构差异、并行计算模型和性能瓶颈分析。适用于需要理解渲染性能、优化 Shader 或设计 GPU 友好数据结构的开发者。

## 一句话结论

GPU 擅长大规模相似任务的并行执行，但受分支发散、随机内存访问和同步影响；优化前应先判断是算力、带宽还是延迟受限。

## 核心概念

### 为什么渲染适合并行

顶点变换、三角形处理、像素着色、纹理采样和光照计算包含大量独立或弱依赖任务，因此 GPU 采用大量并行执行单元，以吞吐量而非单线程低延迟为主要目标。

GPU 并行效率会受到以下因素影响：

- 分支发散
- 随机内存访问
- 线程同步
- 原子操作
- 过大的中间数据

### CPU 与 GPU 的分工

| 特性 | CPU | GPU |
|---|---|---|
| 核心数量 | 少（4-64） | 多（数百-数千） |
| 单核性能 | 高 | 低 |
| 控制流 | 复杂分支、任务调度 | 简单、规则 |
| 内存访问 | 随机、低延迟 | 连续、高带宽 |
| 适用任务 | 场景管理、资源加载、逻辑 | 顶点/像素/计算线程批处理 |

性能瓶颈可能来自 CPU 提交、GPU 执行、内存带宽或同步，不能只看单个 Shader。

## 工作原理

### GPU 执行模型

GPU 通常以成组线程（warp/wavefront）执行相似指令。同一组线程走不同分支时，硬件可能分别执行多个路径，导致有效吞吐下降。

#### SIMD/SIMT 与分支发散

- **SIMD**：单指令多数据，所有线程执行相同指令。
- **SIMT**：单指令多线程，允许线程有独立状态，但分支会导致串行化。

重点检查：

- 动态分支（依赖材质或像素数据）
- 循环次数不一致
- Shader 变体膨胀
- 寄存器压力和低占用率

#### 内存层次

需要区分：

- **寄存器**：最快，每个线程私有
- **共享/本地存储**：线程组共享，需要显式管理
- **缓存**：L1/L2，自动管理
- **纹理缓存**：针对纹理访问优化
- **全局显存**：最慢，带宽受限

连续、合并、可预测的访问有利于吞吐；随机访问和过大的工作集容易导致缓存失效。

### GPU 性能判断

#### 算术受限

指令、特殊函数或循环太多。

- 表现：降低分辨率后帧率提升不明显
- 验证：GPU Profiler 查看 ALU 利用率
- 处理：减少指令、简化数学、使用近似

#### 带宽受限

纹理、顶点、Render Target 或中间缓冲读写过多。

- 表现：降低分辨率后帧率明显提升
- 验证：GPU Profiler 查看内存带宽
- 处理：压缩纹理、减少 RT 切换、优化数据布局

#### 延迟受限

随机访问导致缓存命中率低，线程无法隐藏内存延迟。

- 表现：占用率低但带宽未饱和
- 验证：GPU Profiler 查看占用率和缓存命中率
- 处理：改善访问模式、增加并行度、预取数据

## 关键数据结构

### 渲染数据设计原则

- 让同一批线程访问相近的数据
- 减少 CPU 与 GPU 之间不必要的数据往返
- 避免每个对象都触发独立状态切换
- 分离频繁更新数据和静态数据
- 为质量、内存和带宽设定明确预算

这些原则会直接影响光源剔除、延迟着色、Clustered Shading、体素 GI 和光照缓存设计。

### 常见数据结构

| 数据结构 | 用途 | GPU 友好性 |
|---|---|---|
| 顶点缓冲区 | 存储几何数据 | 高（连续访问） |
| 索引缓冲区 | 复用顶点 | 高（减少带宽） |
| 纹理 | 存储 2D/3D 数据 | 高（缓存优化） |
| 结构化缓冲区 | 通用数据存储 | 中（需要合并访问） |
| 原子计数器 | 线程间同步 | 低（串行化） |

## 开发者如何判断

### 初步判断 CPU/GPU 瓶颈

1. 降低渲染分辨率，观察帧率变化
   - 明显提升：GPU 受限
   - 变化不大：CPU 受限或垂直同步
2. 关闭复杂 Shader，观察帧率变化
   - 明显提升：Shader 受限
   - 变化不大：带宽或 CPU 受限
3. 使用 GPU Profiler 查看占用率和带宽

### Shader 优化检查清单

- [ ] 是否有不必要的动态分支？
- [ ] 循环次数是否一致？
- [ ] 纹理采样是否过多？
- [ ] 数学运算是否可以简化？
- [ ] 精度是否合适（half vs float）？
- [ ] 是否有冗余计算？

### 带宽优化检查清单

- [ ] 纹理格式是否压缩？
- [ ] Render Target 格式是否合适？
- [ ] 是否有不必要的 RT 切换？
- [ ] 顶点属性是否精简？
- [ ] 是否使用了 Mipmap？

## 实现方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 减少指令 | 直接降低算力需求 | 可能影响质量 | 算术受限 |
| 纹理压缩 | 减少带宽，提高缓存命中 | 可能有质量损失 | 带宽受限 |
| 减少 RT 切换 | 减少带宽和状态切换 | 需要重组 Pass | 带宽受限 |
| 合并 Draw Call | 减少 CPU 提交 | 需要批处理逻辑 | CPU 受限 |
| 使用计算着色器 | 灵活，可复用数据 | 需要手动管理 | 复杂后处理 |
| 异步计算 | 提高 GPU 利用率 | 需要仔细同步 | 独立任务 |

## 常见问题与排查

| 现象 | 可能原因 | 验证方式 | 处理方向 |
|---|---|---|---|
| 帧率低，降低分辨率无明显提升 | CPU 受限 | 查看 CPU Profiler | 优化逻辑、减少 Draw Call |
| 帧率低，降低分辨率明显提升 | GPU 受限 | 查看 GPU Profiler | 优化 Shader、减少带宽 |
| Shader 慢，但 ALU 利用率低 | 带宽或延迟受限 | 查看内存带宽和占用率 | 优化访问模式、压缩纹理 |
| 占用率低，带宽未饱和 | 延迟受限 | 查看缓存命中率 | 改善访问模式、增加并行度 |
| 分支发散严重 | 动态分支 | 查看分支统计 | 使用静态分支、重组线程 |
| 寄存器压力高，占用率低 | 寄存器溢出 | 查看寄存器使用 | 简化 Shader、减少变量 |

## 术语速查

| 中文 | English | 说明 |
|---|---|---|
| 算术受限 | ALU Bound | 指令或数学运算过多 |
| 带宽受限 | Bandwidth Bound | 内存读写过多 |
| 延迟受限 | Latency Bound | 缓存未命中，线程等待 |
| 分支发散 | Divergence | 线程组内走不同分支 |
| 占用率 | Occupancy | 活跃线程数与最大线程数的比例 |
| 合并访问 | Coalesced Access | 连续线程访问连续内存 |
| 纹理缓存 | Texture Cache | 针对纹理访问优化的缓存 |
| 结构化缓冲区 | Structured Buffer | 通用数据存储缓冲区 |
